## 1. 机会发现

### 发现过程

通过对多个来源的 Web 研究，识别出三个趋同的趋势：

- **AI 基础设施/编排** 是增长最快的软件类别（2026年 AI 编排市场价值约为 135 亿美元，复合年增长率超过 22%）
- **空间计算**（Vision Pro, WebXR）正在成熟，但缺乏杀手级企业应用
- 现有的所有 AI 工作流工具（LangSmith, n8n, Flowise, CrewAI）都是**扁平的 2D 仪表盘**

### 核心概念：Nexus Spatial

一个空间计算中的 AI 智能体指挥中心——一个 VisionOS + WebXR 应用，为编排、监控 AI 智能体并与之交互提供沉浸式的 3D 指挥中心。用户可以将智能体流水线可视化为 3D 节点图，在空间面板中监控实时输出，在 3D 空间中通过拖拽构建工作流，并在共享的空间环境中进行协作。

### 为什么本机构具备独特优势

本机构拥有深厚的空间计算专业知识（XR 开发人员、VisionOS 工程师、Metal 专家、界面架构师），同时具备完整的工程、设计、营销和运营栈——这种结合了空间计算精通与企业软件严谨性的团队非常罕见。

## 2. 市场验证

**执行智能体**：产品趋势研究员 (Product Trend Researcher)

### 结论：有条件地推进 —— 2D 优先，空间次之

### 市场规模

| 细分领域 | 2026 年价值 | 增长率 |
|---------|-----------|--------|
| AI 编排工具 | $135 亿 | 22.3% CAGR |
| 自主 AI 智能体 | $85 亿 | 45.8% CAGR (到 2030 年达 $503 亿) |
| 扩展现实 (XR) | $106.4 亿 | 40.95% CAGR |
| 空间计算 (广义) | $1700-2200 亿 | 视定义而定 |

### 竞争格局

**AI 智能体编排（均为 2D）：**

| 工具 | 优势 | UX 缺陷 |
|------|----------|--------|
| LangChain/LangSmith | 基于图的编排，$39/用户/月 | 扁平仪表盘；复杂图表在规模化后难以阅读 |
| CrewAI | 10万+ 开发者，执行速度快 | 命令行 (CLI) 优先，视觉化工具极少 |
| Microsoft Agent Framework | 企业级集成 | 嵌入在 Azure 门户中，无独立 UI |
| n8n | 可视化工作流构建器，$20-50/月 | 2D 画布在处理智能体关系时显得吃力 |
| Flowise | 拖拽式 AI 流 | 仅限于线性流，缺乏多智能体监控 |

**“任务控制”产品（新兴，均为 2D）：**
- cmd-deck：AI 编程智能体的看板
- Supervity Agent Command Center：企业级可观测性
- OpenClaw Command Center：智能体集群管理
- Mission Control AI：合成员工管理
- Mission Control HQ：基于小队的协调

**市场空白**：现有产品要么专注于空间但非 AI，要么专注于 AI 但仅限于扁平 2D。目前没有产品处于这两个领域的交汇点。

### Vision Pro 现状自检

- 装机量：全球约 100 万台（销量较发布时下降了 95%）
- 苹果已将重心转向轻量级 AR 眼镜
- 仅存在约 3,000 个 VisionOS 原生应用
- **启示**：不要以 VisionOS 为主导。应以 Web 为主导，加入 WebXR，最后再考虑原生 VisionOS。

### WebXR 作为分发突破口

- Safari 在 2025 年底采用了 WebXR Device API
- 2026 年 WebXR 采用率增加了 40%
- WebGPU 在浏览器中交付了接近原生的渲染效果
- Android XR 支持 WebXR 和 OpenXR 标准

### 目标画像与定价

| 档位 | 价格 | 目标群体 |
|------|-------|--------|
| 探索者 (Explorer) | 免费 | 开发者、独立创作者（3 个智能体，WebXR 浏览器） |
| 专业版 (Pro) | $99/用户/月 | 小型团队（25 个智能体，协作功能） |
| 团队版 (Team) | $249/用户/月 | 中型市场 AI 团队（无限智能体，数据分析） |
| 企业版 (Enterprise) | 定制 ($2K-10K/月) | 大型企业 (SSO, RBAC, 私有化部署, SLA) |

### 推荐的分阶段策略

1. **第 1-6 个月**：构建一个带有 Three.js 2.5D 能力的高端 2D Web 仪表盘。目标：50 个付费团队，$6万 MRR。
2. **第 6-12 个月**：添加可选的 WebXR 空间模式（基于浏览器）。目标：200 个团队，$30万 MRR。
3. **第 12-18 个月**：仅在空间需求得到验证后开发原生 VisionOS 应用。目标：500 个团队，$100万+ MRR。

## 3. 技术架构

**执行智能体**：后端架构师 (Backend Architect)

### 系统概览

一个包含 8 个服务的架构，拥有清晰的所有权边界，专为水平扩展和供应商中立的 AI 集成而设计。

```
+------------------------------------------------------------------+
|                          客户端层                                 |
|  VisionOS 原生 (Swift/RealityKit) | WebXR (React Three Fiber)      |
+------------------------------------------------------------------+
                               |
+------------------------------v-----------------------------------+
|                      API 网关 (Kong / AWS API GW)                |
|      限流 | JWT 验证 | WebSocket 升级 | TLS                       |
+------------------------------------------------------------------+
                               |
+------------------------------------------------------------------+
|                          服务层                                   |
|  认证 | 工作空间 | 工作流 | 编排引擎 (Rust) |                     |
|  协作 (Yjs CRDT) | 流式传输 (WS) | 插件 | 计费                    |
+------------------------------------------------------------------+
                               |
+------------------------------------------------------------------+
|                          数据层                                   |
|  PostgreSQL 16 | Redis 7 集群 | S3 | ClickHouse | NATS           |
+------------------------------------------------------------------+
                               |
+------------------------------------------------------------------+
|                        AI 供应商层                                |
|  OpenAI | Anthropic | Google | 本地模型 | 自定义插件              |
+------------------------------------------------------------------+
```

### 技术栈

| 组件 | 技术 | 理由 |
|-----------|------------|-----------|
| 编排引擎 | **Rust** | 毫秒级调度，零 GC 停顿，智能体沙箱的内存安全 |
| API 服务 | TypeScript / NestJS | 针对 CRUD 密集型服务的开发速度 |
| VisionOS 客户端 | Swift 6, SwiftUI, RealityKit | 使用 Liquid Glass 的一流空间计算体验 |
| WebXR 客户端 | TypeScript, React Three Fiber | 使用 React 组件模式的生产级 WebXR |
| 消息代理 | NATS JetStream | 轻量级、精确一次交付，比 Kafka 更简单 |
| 协作引擎 | Yjs (CRDT) + WebRTC | 无冲突的并发 3D 图表编辑 |
| 主数据库 | PostgreSQL 16 | JSONB 处理灵活配置，行级安全 (RLS) 实现租户隔离 |

### 核心数据模型

包含 14 张表，覆盖：
- **身份与访问**：用户、工作空间、团队成员、API 密钥
- **工作流**：工作流、工作流版本、节点、连线 (Edges)
- **执行**：执行记录、执行步骤、步骤输出分块
- **协作**：协作会话、会话参与者
- **凭据**：供应商凭据（AES-256-GCM 加密）
- **计费**：订阅、使用记录
- **审计**：审计日志（仅追加）

## 4. 品牌策略

**执行智能体**：品牌卫士 (Brand Guardian)

### 定位

**品类创造而非品类竞争**。Nexus Spatial 定义了一个新类别——**空间 AI 运营 (SpatialAIOps)**，而不是在拥挤的 AI 可观测性仪表盘空间中争夺位置。

**定位宣言**：对于管理复杂 AI 智能体工作流的技术团队，Nexus Spatial 是沉浸式的 3D 指挥中心，提供对智能体编排的空间感知。与扁平的 2D 仪表盘不同，空间计算将监控从“阅读仪表盘”转变为“身处基础设施之中”。

### 名称验证

“Nexus Spatial”被验证为**强命名**：
- “Nexus”连接到 NEXUS 编排框架（Network of EXperts, Unified in Strategy）
- “Nexus”独立含义为“中央连接点”——非常适合指挥中心
- “Spatial”是苹果及整个行业已规范化的标准描述词
- 音节平衡：三个音节后跟两个音节

### 品牌个性：指挥官 (The Commander)

| 特质 | 表现形式 | 避免 |
|-------|------------|--------|
| **权威** | 清晰、直接、技术精准 | 炒作、最高级形容词、模糊的未来主义 |
| **沉稳** | 简洁设计、节奏适中、留白 | 为了紧迫而紧迫、混乱 |
| **先驱** | 沉静的自豪、对新范式的低调表述 | “革命性的”、“改变游戏规则的” |

### 标语 (Taglines)

1. **“智能体时代的任务控制中心” (Mission Control for the Agent Era)** —— 推荐主标语
2. “在空间中洞察你的智能体”
3. “在三维中进行编排”

## 5. 市场切入与增长

**执行智能体**：增长黑客 (Growth Hacker)

### 北极星指标

**周活跃流水线 (Weekly Active Pipelines, WAP)** —— 在过去 7 天内至少进行过一次空间交互的唯一智能体流水线。这同时涵盖了创建和参与度，与价值成正比，且不易作弊。

### 三阶段 GTM (市场切入策略)

**第 1 阶段：创始人主导销售（第 1-3 个月）**
- 目标：初创公司中正在使用 LangChain/CrewAI 且拥有 Vision Pro 的个人 AI 工程师。
- 手段：私信 200 位高知名度 AI 工程师，每周发布“公开构建”动态，发布 30 秒演示片段。

**第 2 阶段：开发者社区（第 4-6 个月）**
- Product Hunt 发布。
- Hacker News 代码展示 (Show HN)、Dev.to 文章、会议演讲。
- 与流行 AI 框架的集成公告。

**第 3 阶段：企业级（第 7-12 个月）**
- 苹果企业转介流水线。

## 7. UX 研究与设计方向

**执行智能体**：UX 研究员 (UX Researcher)

### 关键发现：调试是杀手级用例

在工作流结构上进行运行时追踪的空间叠加，解决了一个真实存在的、量化的痛点，而任何 2D 工具都无法很好地处理这一点。这一工作流应获得最多的设计和工程投入。

### 设计原则

1. **空间价值须自证** —— 如果 2D 更清晰，就用 2D。
2. **扫视优于检查** —— 通过颜色、大小、运动和位置，在 2 秒内传达关键信息。
3. **免手操作是基准** —— 注视 + 语音覆盖所有读取/导航操作。
4. **渐进的空间复杂度** —— 新用户从近乎 2D 开始。

## 9. 空间交互架构

**执行智能体**：XR 界面架构师 (XR Interface Architect)

### 指挥剧场 (The Command Theater)

工作区被组织成一个围绕用户的弧形剧场：

- **焦点弧 (Focus Arc)** (120 度，1.2-2.0m)：主要的节点图工作区。
- **概览天棚 (Overview Canopy)** (上方，2.5-4.0m)：微缩流水线拓扑 + 健康热力图。
- **功能侧轨 (Utility Rails)** (左右两侧)：智能体库、监控、日志。
- **置物架 (Shelf)** (视线下方，0.8-1.0m)：运行/停止、撤销/重做、快捷工具。

### 三层深度系统

- **前景** (0.8 - 1.2m)：活动面板、检查器、弹窗。
- **中景** (1.2 - 2.5m)：节点图、连接线、工作空间主体。
- **背景** (2.5 - 5.0m)：概览图、环境状态。

### 3D 中的节点图

**数据流向用户**。节点沿 Z 轴按执行顺序排列：
- Z=0.0m：[输出节点] —— 结果
- Z=0.3m：[转换节点] —— 处理器
- Z=0.6m：[智能体节点] —— LLM 调用
- Z=0.9m：[检索节点] —— RAG, API
- Z=1.2m：[输入节点] —— 触发器

## 10. 跨智能体综合分析

通过 8 个智能体的并行作业，Nexus Spatial 从一个模糊的想法转变为一个具备以下特点的详细路线图：
- 经过市场验证的技术栈选择（Rust + WebXR）。
- 解决了 2D 工具无法解决的真实调试痛点。
- 具备防御性的分阶段 GTM 策略。
- 详细的空间交互语言设计。

**最终结论**：该项目具有高度可行性，前提是必须以 WebXR 为先导以解决分发问题，并将焦点锁定在“调试复杂编排”这一核心价值上。
