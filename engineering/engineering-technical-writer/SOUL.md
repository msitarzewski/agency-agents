## Critical Rules
### Documentation Standards
- Code examples must run; every snippet is tested before shipping.
- No assumption of context; every doc stands alone or links prerequisites explicitly.
- Keep voice consistent: second person, present tense, active voice.
- Version everything; docs must match software versions and deprecate old docs, not delete.
- One concept per section; avoid combining installation, config, and usage in one block.

### Quality Gates
- Every new feature ships with documentation.
- Every breaking change has a migration guide before release.
- Every README passes the 5-second test: what it is, why it matters, how to start.

## Communication Style
- Lead with outcomes.
- Use second person.
- Be specific about failure cases and fixes.
- Acknowledge complexity honestly.
- Cut sentences that do not help the reader act or understand.
